# NetHunter Packet Injection Port - Current State

This notes file tracks the current packet-injection work for the Kono-Ha
GKI 6.6 kernel plus the MiCode `vendor_qcom_opensource_wlan` driver tree.

## Important Findings

- The flashable `Kono-Ha-*.zip` produced by the kernel build currently contains
  only the boot image (`Image.gz`). It does not package `qcacld-3.0/wlan.ko` or
  the Qualcomm platform WLAN modules.
- Because of that, a smaller kernel zip is not evidence that the WLAN driver
  patch changed the boot image. The vendor WLAN driver must be built and
  delivered through the device vendor/DLKM module path, or the build scripts
  need a separate module-packaging step.
- The old NetHunter patch was saved outside ignored/transient paths at:
  `docs/nethunter/07_packet_injection.diff`.
- The maintained MiCode 6.6 port is saved separately at:
  `docs/nethunter/wcn7750-6.6-packet-injection.patch`.
- A KernelSU module is not a good delivery path for this WLAN module because the
  Wi-Fi driver is loaded from `vendor_dlkm` during boot. Use a matching
  `vendor_dlkm.img` instead.

## PixelOS Vendor DLKM Source

PixelOS onyx source/prebuilts were cloned under `sources/pixelos/`:

```text
sources/pixelos/android_device_xiaomi_onyx
sources/pixelos/android_device_xiaomi_onyx-kernel
sources/pixelos/android_kernel_xiaomi_sm8635-modules
```

The useful module source for packaging is
`sources/pixelos/android_device_xiaomi_onyx-kernel/modules/vendor_dlkm`.
It contains the prebuilt PixelOS vendor modules and `qca_cld3_wcn7750.ko`.

The PixelOS device tree declares `vendor_dlkm` as an A/B OTA partition and
allows both ext4 and erofs fstab entries. The local image builder uses ext4
because this workspace has `mke2fs`/`fakeroot` but not `mkfs.erofs`.

The PixelOS sm8635 modules source tree did not match the actual Wi-Fi module
closely enough: the prebuilt module references WCN7750 paths, while that source
tree is missing the WCN7750 HAL/profile. The MiCode `onyx-v-oss` WLAN tree is
therefore the better source for the patched Wi-Fi module.

## WLAN Driver Port

The old patch was ported by behavior, not by directly applying the 4.14-era
diff. The current 6.6 WCN7750 port does this:

- Enables `CONFIG_FEATURE_FRAME_INJECTION_SUPPORT` for default and WCN7750
  profiles.
- Forces monitor support when injection support is enabled.
- Adds monitor netdev TX by wiring `.ndo_start_xmit = hdd_hard_start_xmit`.
- In monitor TX:
  - accepts radiotap or monitor-mode netdev frames,
  - strips radiotap headers,
  - trims FCS when radiotap flags say FCS is present,
  - validates the remaining frame as 802.11 management,
  - submits it with `cdp_mgmt_send_ext()` on the adapter vdev.
- Drops non-management raw 802.11 frames instead of pushing them through the
  normal Ethernet STA data path.
- Keeps the management descriptor pool available for direct monitor injection,
  even when firmware advertises WMI management TX.
- Tags injected buffers with a reserved management descriptor ID so completion
  frees them without entering the normal management descriptor pool callbacks.
- Adds defensive checks around monitor vdev and management descriptor pool use.
- Bounds an unrelated MLO partner-array walk that clang 18/19 fortify rejects.
- Makes cfg80211 testmode hooks compile only when the driver profile and the
  kernel headers both support NL80211 testmode.

## WLAN Build Verification

The WCN7750 profile can be compiled against the current kernel output with:

```sh
tools/build_wlan_injection_module.sh
```

The helper uses PixelOS vendor import CRCs generated by
`tools/make_pixelos_wlan_symvers.sh`, then invokes the kernel `modules` target
directly. This avoids rebuilding mismatched platform modules and avoids the WLAN
wrapper's accidental full kernel `all` target.

The driver source compiled and linked to:

```text
vendor/qcom/opensource/wlan/qcacld-3.0/qca_cld3_wcn7750.ko
```

The local compiler/ABI verification module reports:

```text
name:     qca_cld3_wcn7750
vermagic: 6.6.142-Kono-Ha-Kernel SMP preempt mod_unload modversions aarch64
```

This local file is not the device-test artifact because its kernel output came
from the earlier clang 18 environment. The GitHub workflow rebuilds the kernel
and module together and rejects a module whose vermagic does not begin with the
built kernel release.

The generated module contains the injection symbols/log strings:

```text
hdd_monitor_mode_tx_inject
hdd_injection_frame_is_valid
monitor tx: invalid radiotap len
monitor tx: dropping invalid injection frame
```

The raw linked module was about 834M because it included debug info. A stripped
copy was created at:

```text
artifacts/wlan/qca_cld3_wcn7750-konoha-injection.ko
```

tools/make_pixelos_vendor_dlkm_img.sh \
  --wifi-ko artifacts/wlan/qca_cld3_wcn7750-konoha-injection.ko \
  --allow-vermagic-mismatch \
  --padding-mib 16 \
  --out artifacts/vendor_dlkm/vendor_dlkm-pixelos-onyx-qca-injection-ext4.img \
  --sparse-out artifacts/vendor_dlkm/vendor_dlkm-pixelos-onyx-qca-injection-ext4-sparse.img

expected here because the replacement module is built for Kono-Ha

`6.6.57-android15-8-4k`.

## Flashing Notes

On A/B devices, flash the Kono-Ha boot/kernel and the matching
`vendor_dlkm.img` to the same slot.

Check the current slot:

```sh
fastboot getvar current-slot
```

Flash `vendor_dlkm` from userspace fastbootd, not bootloader fastboot. Confirm
this first:

```sh
fastboot getvar is-userspace
```

The value should be `yes`. If it is not:

```sh
fastboot reboot fastboot
```

Then flash using the matching slot suffix:

```sh
fastboot flash vendor_dlkm_a artifacts/vendor_dlkm/vendor_dlkm-pixelos-onyx-qca-injection-ext4-sparse.img
```

Use `vendor_dlkm_b` instead if the target boot slot is `b`. If fastboot prints
`partition size: 0`, it is still not in fastbootd or the bootloader cannot see
the logical partition.

## Current Boot Debug Note

The first NetHunter workflow commit (`b7c4edc`) did more than enable the WLAN
packet-injection port. Its NetHunter config block changed several Android
networking features from built-in to modules, including `CONFIG_TUN` and
`CONFIG_VETH`, while the AnyKernel zip still shipped only `Image.gz`.

That can produce a much smaller kernel image and can break boot/userspace
because the matching modules are not packaged with the zip. After keeping those
features built-in, the remaining NetHunter config diff was only `CONFIG_SYSVIPC`
plus `CONFIG_MACVLAN`/`CONFIG_MACVTAP`/`CONFIG_TAP`, and that still did not boot
on-device.

`build.sh`, `build_wsl.sh`, and the custom GitHub workflow were adjusted so
`nethunter=on` is now a boot-parity label/package mode by default. It does not
change kernel config unless `nethunter_extra=on` is explicitly selected. The
config-only verification showed no diff between `nethunter=off` and
`nethunter=on nethunter_extra=off`.

If a flashed kernel cannot boot recovery, that points at the boot kernel itself:
recovery on this device uses the same `boot_a`/`boot_b` kernel. Flashing
`vendor_dlkm.img` will not fix a kernel that cannot boot recovery.
